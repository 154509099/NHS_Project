三、任务中台改造清单

以下为中台（云机汇）侧需完成的改造项，全部要求 6 月底交付。

涉及现有接口概览

接口编号接口名称方向本次变动
A10 门店任务售出办单数据同步 渠道（小河马）→ 中台 ⚠️ 字段改造
A1 任务信息同步 达生 → 中台 不变（复用达生任务主数据）

核心说明：小河马商城订单通过 A10 接口将售出办单数据同步给中台。本次改造的核心是让 A10 支持无罐底码的订单透传，并新增退货方向的两个接口完成扣减闭环。

改造1：A10 办单数据同步接口调整

改造点具体内容
罐底码改为非必填 A10 接口入参中 罐底码 字段由必填改为非必填。小河马商城订单传入时不携带罐底码，仅传 SKU、数量、订单金额等业务字段。
新增渠道类型字段 A10 入参及中台订单表新增 channelType（渠道类型）字段，枚举值：XHM_MALL（小河马商城）/ STORE_NORMAL（普通门店办单）。商城订单标识为特殊渠道。
分类校验逻辑 中台接收 A10 数据后，按 channelType 分流处理：
• XHM_MALL：走白名单通道，跳过罐底码有效性校验、新老客校验、会员码校验，传入即视为有效订单
• STORE_NORMAL：保持现有校验逻辑不变
推送达生时去罐底码 中台将商城订单推送达生时，订单数据中不携带罐底码字段，达生按 SKU + 数量 + 佣金金额进行积分发放。
佣金由达生侧计算 佣金计算规则（各 SKU 佣金比例）由达生侧配置维护。小河马通过 A10 传入时仅传 SKU 和数量，中台透传给达生，达生按规则自行计算佣金后发放。
接口幂等性 A10 以 订单号 + eventType（正向） 作为幂等键，中台侧记录已处理事件流水，防止小河马重复推送导致积分多发。

  • panda："这个订单就是个特殊订单了，相当于走了一个叫什么白名单通道了" → 锦鹏："那他知道这样子就不会做任何校验，就只要传过来就其实就算他是有效了"（11:37-11:47）

改造2：退货通知接口及达生扣减链路（新增）

2.1 小河马 → 中台：退货通知接口（中台侧接收）

改造点具体内容
接口定义 中台新增接收小河马退货通知的 HTTP 接口。消费者在小河马商城发起退货后，小河马调用此接口推送退货信息。
入参字段 orderId（订单号）、returnType（整单退/部分退）、returnItems（退货明细：SKU + 退货数量 + 退货金额）、returnTime（退货时间）、refundStatus（退款状态）
部分退货支持 接口需同时支持整单退和部分退两种模式。部分退时 returnItems 传入具体退款的 SKU 及数量，中台按比例计算应扣佣金金额。
幂等性设计 以 orderId + returnType + returnItems hash 作为幂等键，中台侧记录退货事件流水，防止小河马重复推送导致多次扣减。
异常重试 小河马侧推送失败时，中台需支持补推（参考现有渠道补推机制），小河马可按中台通用规范定时重试推送。

2.2 中台 → 达生：积分扣减通知接口（中台侧发起）

改造点具体内容
接口定义 中台新增向达生发起积分扣减的 HTTP 接口。中台收到退货通知并校验通过后，调用达生扣减接口。
入参字段 orderId（原始订单号）、storeAccountId（门店账户ID）、deductAmount（扣减金额）、deductReason（扣减原因=退货）、returnRef（关联退货事件ID）
扣减时机约束 达生扣减须在结算周期内完成：资金处于冻结/未提现状态时可扣减；若已提现，达生返回扣减失败，中台记录异常并提示人工处理（首期约束为仅允许未提现扣减）。
达生响应处理 中台根据达生返回状态码分类处理：成功 → 记录扣减流水；失败（已提现）→ 标记异常待人工；失败（系统错误）→ 定时重试。

改造3：中台数据记录与对账

改造点具体内容
全链路事件流水 中台新增商城订单事件流水表，记录每条订单全生命周期：A10接收 → 推送达生 → 积分发放 → 退货通知 → 积分扣减，每步记录时间戳和操作人/系统，形成可追溯链。
对账基准 以小河马商城订单状态为对账基准。中台定期（建议每日 T+1）与达生侧核对三组数据：①当日发放总额 ②当日扣减总额 ③当前门店待提现余额。不平项生成差异报告。
异常事件处理 达生侧积分发放失败或扣减失败时，中台记录异常日志，支持自动重试（最多 3 次，间隔递增），3 次后仍失败则转为人工介入工单。
补推能力复用 参考现有渠道补推机制（中台→渠道 通用规范），正向 A10 接收和逆向退货通知均支持补推，确保数据不丢。
核销数据输出 中台需支持按大区、门店维度输出：分配盒数、实际售出盒数、已兑换/提现盒数，供雀巢和小河马做业务核销（panda 明确提出的长期需求）。

3. 商品售后与投诉处理流程 🟡 中

六、专业建议

行业参考

  1. 代销分佣模式常见做法：目前主流电商平台（淘宝客、京东联盟、美团等）的分佣模式均基于 CPS（按成交付费），通过联盟平台或开放平台的 API 回传订单信息，分佣系统独立于主交易系统。小河马方案本质是"私域 + 线上代销"的复合模式，可以借鉴：
    • 订单回传后由独立结算系统处理佣金，不与主交易系统耦合
    • 退货扣佣通常通过佣金结算周期内的对冲实现（而非实时代扣）
  2. 无溯源码的订单分佣：行业中有类似场景（如美团外卖门店分佣），通过订单号+商品SKU+数量来完成分账，不依赖实物溯源码。中台改造成"溯源码可选"是合理方向。

坑点提醒

  1. 部分退货的佣金计算精度问题：如果一单多罐，退部分罐，佣金按比例扣减时可能存在精度损失（如 0.01 元差额），需要提前定义取整规则（向上取整/向下取整/四舍五入），避免账务不平。
  2. 达生积分提现的风控：门店通过达生提现意味着门店需要在达生体系内完成实名认证。门店老板是否愿意在非自有系统中完成实名和绑卡？这是一个潜在的推行阻力。
  3. 资金池对账复杂度：小河马（汇付天下）→ 中台 → 达生，三套资金/积分体系，对账链条长。建议：
    • 定期（如每日/每周）出一份三端对账报表
    • 明确哪个端作为"对账基准"（建议以小河马商城订单状态为准）

技术约束

  1. 汇付天下分账能力：需确认汇付天下是否支持部分退款自动分账回退（即退款时同时从收款方账户扣回已分账金额）。如果汇付不支持，则退货时需要小河马自行处理资金回转逻辑。
  2. 达生积分扣减能力：需确认达生是否支持"积分已发放但未提现"状态下的扣减操作。如果达生不支持，需要走"撤销任务"而非"扣减积分"的逻辑。
  3. 接口幂等性：订单推送和退货通知接口需保证幂等，防止重复推送导致门店积分多发或多扣。
