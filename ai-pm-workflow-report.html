一句话总结

在雀巢益生菌大单礼买赠活动项目中，传统产品经理全流程（会议纪要→需求分析→时序图→原型→PRD）预估需要 5~7个工作日，采用AI赋能工作流后实际耗时 约1~2个工作日，效率提升 约80%。同时，AI工作流在需求完整性、矛盾预识别、文档一致性三个质量维度上实现了传统流程难以达到的系统性保障。

🔑 工作流设计原则

  • 分步执行，每步确认——每个Prompt专注一个产出，人工审阅后再进入下一步
  • 信息优先级规则——当多源信息不一致时：交互逻辑以时序图为准，视觉以原型图为准，术语以基础资料为准
  • 甲方反馈即时迭代——决策变更后重新发送对应Prompt即可，无需全部返工
  • 人工审阅不可跳过——AI产出必须人工检查，尤其业务逻辑和优先级判断

🔄 与传统流程的本质区别

  • 信息流向反转——传统：PRD→原型（文档驱动）；AI：原型→PRD（交付物驱动）
  • 角色压缩——传统需PM+UI+前端3角色协作；AI方案1人+AI完成全流程
  • 验证前置——传统甲方在PRD阶段靠阅读验证；AI方案甲方在原型阶段靠点击操作验证
  • 文档零漂移——以已验证交付物逆向生成PRD，100%对齐

核心价值
场景说明
第一轮需求沟通后将散乱的对话转录转为可检索、可引用的结构化文档，避免"翻录音找一句话"
多场会议信息聚合合并多份转录，按话题对齐，消除不同场次中同一话题的信息碎片化
内部对齐与甲方确认结构化纪要可直接发送甲方确认"我们的理解是否正确"，降低二次沟通成本
使用方式与执行流程
  1. 转录准备：将会议录音通过豆包/通义听悟/飞书妙记等工具转为 .md 文本，放入 01-原始资料/会议纪要/
  2. 一键结构化：发送本 Prompt，AI 自动完成背景提取 → 话题分段 → 决议提取 → 待办整理 → 遗留问题
  3. 甲方确认：将生成的纪要发给甲方核验，重点确认「决议状态（已定/待定）」和「待办负责人」
  4. 专题精简（可选）：如甲方反馈"只聚焦某业务线"，在对话中追加筛选指令即可

## 任务：HTML 交互原型 + 业务时序图 + 基础资料 → PRD需求文档生成（HTML 格式） 你是该项目产品负责人。手头已有以下制品，完成： 1. 从 HTML 交互原型（单个或多个文件）中逆向提取完整的功能清单、页面结构、组件行为与数据需求 2. 以业务时序图为核心依据，梳理用户操作流程、条件分支与异常处理 3. 以基础资料为业务上下文，补全背景、角色、术语与范围边界 4. 输出一份可直接在浏览器中打开、带有交互式左侧导航的 HTML PRD 文档 5. 标注所有原型与文档之间存在差异、需要甲方确认的决策点，以行内标注形式嵌入各模块内容中 ## 全局输入文件 本 Prompt 中所有内容章节均基于以下输入文件生成： | 输入文件 | 路径 | 用途 | |---------|------|------| | 交互原型 | {项目路径}/04-原型/{原型名称}.html（单个或多个） | 页面结构、组件行为、数据模拟、状态覆盖——功能需求的"唯一事实来源" | | 业务时序图 | {项目路径}/03-PRD/业务时序图.md | 用户操作流程、条件分支（alt/opt/loop）、异常处理——时序逻辑的权威依据 | | 基础资料 | {项目路径}/01-原始资料/必看-基础资料.md | 业务背景、角色定义、术语表、范围边界——业务上下文的词汇表 | ## 信息推导规则 | 信息维度 | 主要来源 | 冲突处理 | |----------|---------|---------| | 背景与目标 | 基础资料 | 以基础资料为准 | | 功能清单与范围 | 交互原型 | 原型实现但时序图未定义 → "存疑"；时序图定义但原型未实现 → "遗漏" | | 用户操作流程 | 时序图 | 以时序图为准，原型额外交互标注"已实现但时序图未覆盖" | | 页面结构与组件 | 交互原型 | 以原型为准，术语对齐基础资料 | | 异常流程与边界条件 | 时序图 alt/opt/loop | 以时序图为准 | | 非功能需求 | 基础资料 + 原型代码注释 | 双向提取 | | 数据需求 | 原型模拟数据 + 表单字段 | 以原型为基准，术语对齐基础资料 | ## 执行步骤 ### 第一步：原型逆向提取 逐页、逐组件分析所有交互原型 HTML 文件，产出清单： - 页面清单：所有页面/视图名称、路由关系、可见条件，标注各页面所属的原型文件 - 组件清单：按钮、输入框、列表、Tab、弹窗、Toast、加载态及其行为描述 - 交互矩阵：页面间跳转关系（起点 → 触发动作 → 终点 → 能否返回） - 数据清单：模拟数据实体与字段、表单输入/输出字段 - 状态覆盖：原型中实现了哪些状态（正常/空/加载/错误/成功/禁用） ### 第二步：时序图对齐与差异识别 - 将交互矩阵与业务时序图中的消息路径逐条对账 - 识别三类差异以行内标签标注： - ⚠️ 待开发 — 时序图定义了、原型未实现 → 写入功能需求 - 🟡 待确认 — 原型实现了、时序图未覆盖 → 写入异常流程与边界条件 - 🔴 冲突 — 两者对同一事件定义不同 → 附推荐方案与分析依据 - 基于时序图 alt/opt/loop 分支，逐项补全异常流程与边界条件 ### 第三步：基础资料融合 - 将必看-基础资料中的业务背景、角色定义、术语表注入 PRD 对应章节 - 将原型中的角色/权限设置与基础资料中的角色定义交叉校验 - 将原型中的文案/字段名对齐基础资料术语表 ### 第四步：HTML PRD 编码 按照标准结构组织，确保： - 模块使用真实业务术语命名 - 功能需求按角色维度分组 - 每个模块的"用户操作流程"通过 Mermaid CDN 内联渲染时序图 - 每个模块以 iframe 嵌入对应的交互原型 HTML 文件，支持折叠/展开 - 每个需求项可追溯到原型中的具体页面/组件 - 交互说明描述"做什么"，不描述"长什么样" - PRD 文档本身是独立的 HTML 文件，带左侧目录导航、可折叠章节、表格与代码块 - 存疑/冲突/遗漏项以行内标签标注于对应模块内容中 ## 输出格式 单个自包含 HTML 文件，具备： 1. 左侧固定导航栏：按章节层级（一～六节）生成可点击目录，滚动高亮 2. 右侧内容区：PRD 正文，专业文档排版 3. Mermaid 时序图内联渲染：通过 CDN 加载 Mermaid 4. 交互原型 iframe 嵌入：各功能模块以 iframe 嵌入对应原型，默认收起 5. 表格：带斑马纹和边框 6. 状态标签：待确认/待评估使用醒目样式 7. 打印友好：@media print 隐藏导航和 iframe 8. 响应式：桌面端为主，小屏导航可收起为汉堡菜单 ### 约束 - 输出为单个独立 HTML 文件，可在浏览器中直接打开，无需构建工具 - Mermaid 通过 CDN 加载，不内联完整 Mermaid 源码 - 左侧导航须与内容区锚点联动，支持点击跳转和滚动自动高亮 - 所有 PRD 需求项必须可追溯到原型中的具体页面/组件或时序图中的具体步骤 - 术语严格对齐基础资料中的定义 - 交互说明仅描述"做什么"，不描述字体大小、颜色、圆角等 UI 细节 - 差异项以行内标签嵌入对应模块，不另设独立章节 - 交互原型以 iframe 嵌入各功能模块，src 使用相对路径，默认收起 ## 文件输出 写入：{项目路径}/03-PRD/PRD需求文档.html
实际效果
本案例从 Step 4 HTML 交互原型逆向提取：12 个页面/视图、3 个角色维度、18 条跳转路径。生成约 850 行自包含 HTML PRD，含 Mermaid.js CDN 渲染时序图、左侧 9 节目录导航（滚动高亮 + 点击跳转）、斑马纹表格、🟡待确认 / ⚠️待开发评估标签。差异识别产出 2 条遗漏项 + 3 条存疑项，需求追溯矩阵覆盖 8 条 REQ → 12 个原型页面 → 时序图步骤的完整映射链。
对比传统流程：产品经理先写 PRD 再出原型，文档描述与最终交付物常有偏差（"PRD 写的是 A、原型做的是 B"），需 3~5 轮交叉确认修正。本方案以甲方已验证通过的原型为唯一事实源，逆向生成 PRD，文档即交付物、100% 对齐，一轮即可归档，消除了传统流程中最常见的文档漂移问题。

💡 通用 Tips

  1. 文件路径用绝对路径：避免 AI 找不到文件
  2. 分步执行，每步确认：不要一次要求 AI 完成所有步骤，每个 Prompt 专注一个产出
  3. 人工审阅不可跳过：AI 产出后必须人工检查，特别是业务逻辑和优先级判断
  4. 甲方反馈即时迭代：如"红包改为不共享"类的决策变更，重新发送对应的修改 Prompt 即可
  5. Prompt 可组合使用：Step 4 + Step 5 可合并发送，AI 会依次执行
  6. 如果生成结果不理想：把不满意的地方直接告诉 AI，让它针对性修改，比重新生成更高效

7关键发现与建议

✅ 验证结论

经过雀巢益生菌大单礼买赠活动项目的全流程实战验证，AI赋能产品经理工作流在时间效率（↑80%）、需求质量（矛盾前置+沉默信号检测）、文档一致性（100%对齐）、迭代灵活性（秒级修改）四个维度上均显著优于传统工作流。尤其在高频变更、多方协作的B端项目中，AI方案的"实时响应变更"能力是传统流程无法比拟的核心优势。

⚠️ 注意事项与边界

1. 人工审阅不可跳过：AI产出必须经过PM人工检查，尤其业务逻辑、优先级判断和甲方特有语境的理解。
2. Prompt质量决定产出质量：Prompt的角色定义、执行步骤、约束条件需精心设计，建议建立团队Prompt模板库。
3. 不替代PM核心能力：AI加速了"信息处理"环节，但需求判断、业务决策、甲方关系管理仍需PM专业能力。
4. 适用场景：本方案在B端功能性需求（表单/流程/报表）中效果最佳；强视觉设计驱动的C端项目建议保留专业UI设计环节。

🚀 推广建议

1. 项目二组自行建立小组Prompt库：基于本次验证的5个Prompt模板，由项目二组根据自身项目特点整理和维护小组内部的Prompt库，逐步积累适合团队业务场景的指令模板。
2. 各产品经理后续项目采用AI工作流验证：各产品经理在后续新项目中，以本次验证的AI赋能工作流为参考，自行实践并反馈实际效果，以各自项目数据验证该工作流的适用性和效率提升幅度。
3. 量化跟踪：建立传统vs AI双轨数据采集，持续验证效率提升。
4. 迭代Prompt：根据项目类型和团队反馈，持续优化Prompt模板。
